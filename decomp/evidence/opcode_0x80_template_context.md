# Opcode `0x80` Template Context Scan

## Goal

Narrow the remaining semantic question around the residual `0x80` files after
the first-burst neighboring-family and residual-context passes.

The previous packet already showed that most of the `42` residual files were
not fresh audio-heavy holdouts.
This pass asks a tighter structural question instead:

- which repeated `0x80` burst templates dominate the residual set
- what those bursts immediately hand off to in the decoded script
- whether the few exceptions behave like genuinely different scenes or just
  alternate-state variants

The raw packet for this pass lives in
[`opcode_0x80_template_scan.json`](opcode_0x80_template_scan.json), generated by
[`../verification/analyze_opcode_0x80_templates.py`](../verification/analyze_opcode_0x80_templates.py)
from
[`opcode_0x80_residual_scan.json`](opcode_0x80_residual_scan.json).

## Corpus result

The residual set stays sharply clustered:

- `36 / 42` files use the paired `80 01 41 80 7F` then `80 01 42 80 7F`
  opener
- `3 / 42` files use the paired `80 01 1D 80 7F` then `80 01 1E 80 7F`
  microcluster
- only `3 / 42` files fall outside those two paired templates

That means `39 / 42` residual files are now explained by just two repeated
double-`0x80` bundles, not by a wide spread of unrelated audio-looking scenes.

## Dominant `0x41/0x42` bundle

The `0x41/0x42` family is now the clearest structural result:

- `36` files total
- `34` contain no sound-family opcode anywhere in the file
- only `2` later grow complete neighboring pairs after the first burst
- only `1` currently has a same-title sibling variant that stages a complete
  pre-burst pair first:
  [`408-The Gallows (Mino Zombie, new chest)`](../../decoded_scripts/1-Wine%20Cellar/408-The%20Gallows%20(Mino%20Zombie,%20new%20chest).txt)
  versus
  [`026-The Gallows`](../../decoded_scripts/1-Wine%20Cellar/026-The%20Gallows.txt)

Its surrounding script shape is also highly repetitive:

- `26` files reach the first burst with the exact lead-in
  `DialogHide` -> `Opcode14` -> `SetHeadsUpDisplayMode(1)`
- `20` files then continue with the same five-opcode post-burst spine:
  `Opcode45` -> `ModelControlViaScript` -> `Opcode45` -> `Opcode43` ->
  `ModelControlViaScript`
- the single most common immediate three-line follow-up appears `9` times:
  `45 18 00` -> `42 07 00` -> `45 0D 00`
- another `7` files keep that same `45`/`42`/`45` shape but vary the first
  `Opcode45` argument (`0x1A`, `0x26`, `0x1C`, or `0x18`)

Representative files:

- [`../../decoded_scripts/1-Wine Cellar/018-The Reckoning Room.txt`](../../decoded_scripts/1-Wine%20Cellar/018-The%20Reckoning%20Room.txt)
- [`../../decoded_scripts/11-Iron Maiden B1/360-The Cauldron.txt`](../../decoded_scripts/11-Iron%20Maiden%20B1/360-The%20Cauldron.txt)
- [`../../decoded_scripts/22-Iron Maiden B2/382-The Eunics' Lot.txt`](../../decoded_scripts/22-Iron%20Maiden%20B2/382-The%20Eunics'%20Lot.txt)

The important cleanup point is what this bundle sits next to.
It repeatedly lands in HUD mode handoff, dialog closure, model-script control,
engine-mode changes, geometry suppression, and return-style room-event cleanup.
That is much stronger evidence for shared room-entry or cutscene scaffolding
than for a hidden direct audio consumer living inside `0x80` itself.

## Smaller `0x1D/0x1E` bundle

The `0x1D/0x1E` family is smaller but still template-like:

- `3` files total
- all `3` contain no sound-family opcode anywhere in the file
- none later grow complete neighboring pairs
- none currently have same-title sibling variants with earlier complete pairs

Two of the three files continue into the same post-burst control path:

- `Opcode5A(1, 1, 1, 0, 0, 15)`
- `ModelControlViaBattleMode(113, 8, 2)`
- `Opcode45(20, 0)`

The third instead runs:

- `SetHeadsUpDisplayMode(0)`
- `DialogShow`
- `DialogText("A door opened somewhere...")`

Representative files:

- [`../../decoded_scripts/2-Catacombs/029-The Last Blessing.txt`](../../decoded_scripts/2-Catacombs/029-The%20Last%20Blessing.txt)
- [`../../decoded_scripts/6-Abandoned Mines B1/271-The Passion of Lovers.txt`](../../decoded_scripts/6-Abandoned%20Mines%20B1/271-The%20Passion%20of%20Lovers.txt)
- [`../../decoded_scripts/6-Abandoned Mines B1/272-The Hall of Hope.txt`](../../decoded_scripts/6-Abandoned%20Mines%20B1/272-The%20Hall%20of%20Hope.txt)

So the smaller family also behaves like event-template choreography around
dialog or battle-mode control, not like a concealed direct sound trigger.

## Singleton and variant holdouts

Only `3` residual files sit outside the two paired families:

- [`021-Room of Rotten Grapes`](../../decoded_scripts/1-Wine%20Cellar/021-Room%20of%20Rotten%20Grapes.txt)
  uses a lone `80 03 41 80 7F` inside a longer splash-screen, screen-effect,
  camera, and model-staging cutscene block, then waits and continues into model
  animation work instead of any immediate sound-family setup
- [`022-Chamber of Fear (after the quake)`](../../decoded_scripts/1-Wine%20Cellar/022-Chamber%20of%20Fear%20(after%20the%20quake).txt)
  uses a lone `80 03 0E 80 7F`, then waits and continues through dialog and
  camera-script flow; this is also one of the two same-title variant cases,
  because [`017-Chamber of Fear`](../../decoded_scripts/1-Wine%20Cellar/017-Chamber%20of%20Fear.txt)
  stages the confirmed `0x85/0x88` pair before its own later `0x80`
- [`025-The Hero's Winehall`](../../decoded_scripts/1-Wine%20Cellar/025-The%20Hero's%20Winehall.txt)
  uses a lone `80 01 42 80 7F` and then immediately returns through
  `Opcode43` plus `Opcode6E`, making it look like a minimal room-state template
  rather than a sound path

These holdouts still do not restore a direct sound reading.
They mostly widen the structural story:
the residual `0x80` baggage can sit inside splash-screen, camera, model, or
room-state templates without proving any audible action at the `0x80` site.

## Safe takeaway

- The residual `0x80` problem is now even more template-heavy than the earlier
  residual-context packet suggested.
- `39 / 42` residual files belong to just two repeated double-`0x80` bundles:
  the dominant `0x41/0x42` opener family and the smaller `0x1D/0x1E`
  microcluster.
- Those bundles repeatedly hand off into HUD, dialog, model-script control,
  engine-mode, geometry, and room-state choreography.
- The few exceptions still look like alternate-state or broader cutscene
  variants, not like fresh proof that `0x80` itself triggers sound.

This does not yet promote a better user-facing name for `0x80`.
It does, however, further retire the old direct-sound interpretation and move
the remaining question toward template provenance:
why these shared scaffolds still carry stubbed `0x80` bursts at all.
